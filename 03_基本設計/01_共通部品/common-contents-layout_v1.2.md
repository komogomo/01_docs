## 1. 目的

本書は、HarmoNet アプリケーション全体に共通する「コンテンツ領域（Content Area）」の構造・余白・階層・スタイルルールを定義することを目的とする。

* AppHeader（ch01_AppHeader_v1.0）および AppFooter（ch04_AppFooter_v1.0）に挟まれた **共通コンテンツ領域** の基準を定める。
* ログイン画面（LoginPage / MagicLinkForm / PasskeyAuthTrigger）を含む全画面で、統一された視覚的バランスとアクセシビリティを実現するための **基本設計** とする。

> 本書は「03_基本設計/01_共通部品」の一部として、各画面・各機能の基本設計が参照する“土台”であり、詳細な Tailwind クラス適用は各詳細設計書側に委ねる。

---

## 2. コンテンツ領域の役割

* Header および Footer の間に配置される主情報表示領域。
* 各画面の中心 UI を構成し、カード・フォーム・一覧・詳細・通知などを内包する。
* 利用者の行動の焦点となる領域であり、UI の一貫性・可読性の観点から最も重要な要素のひとつである。

---

## 3. 構造モデル（論理階層）

| 階層 | 要素名       | 説明                                                         |
| -- | --------- | ---------------------------------------------------------- |
| 1  | `<main>`  | 画面ごとの主コンテンツを包括するラッパー。背景色はアプリ全体のベースカラーに従う（原則 `bg-gray-50`）。 |
| 2  | コンテンツコンテナ | 画面幅の制御・左右余白を担当するラッパー。`max-w` と `px` によるレイアウトを担う。           |
| 3  | セクションカード  | 機能ブロック単位のカード（例：ログインフォーム、掲示板一覧、予約フォーム）。角丸・影を持つ。             |
| 4  | コンテンツ本体   | 投稿内容・入力フォーム・一覧行など、実際の情報を表示する要素。                            |

実装時の要素名（`<section> / <article>` 等）は画面設計に応じて選択するが、上記 1〜4 の責務と階層構造を維持すること。

---

## 4. レイアウト寸法

### 4.1 コンテンツ幅と余白

* コンテンツ最大幅：`max-w-3xl`〜`max-w-5xl` を基本とし、ログイン画面など単一フォーム中心の画面は `max-w-md`〜`max-w-lg` を推奨する。
* 画面左右余白：

  * モバイル：`px-4`
  * タブレット：`md:px-8`
  * デスクトップ：`lg:px-12`
* コンテンツ上下マージン：

  * 共通基準：`py-6`
  * 初回ビューで余白を広くとる画面（例：ログイン、トップ）：`md:py-10`

### 4.2 カード間のマージン

* セクションカード同士の縦方向の間隔：`space-y-6` を基本とする。
* 画面によって密度を上げる場合も、`space-y-4` を下限とし、それより詰めない。

## 5. タイポグラフィ指針

| 要素         | スタイル                            | フォント       | 備考                  |
| ---------- | ------------------------------- | ---------- | ------------------- |
| 見出し（h1〜h3） | `font-semibold text-gray-900`   | BIZ UDゴシック | 各画面タイトル・セクション見出しに使用 |
| 本文         | `text-gray-800 leading-relaxed` | BIZ UDゴシック | 一般的な説明文・本文          |
| 補足・説明文     | `text-gray-500 text-sm`         | BIZ UDゴシック | ツールチップ・補助説明・注意書き    |

フォントサイズの具体値（`text-base` / `text-lg` 等）は、HarmoNet デザインシステムの定義に従い、各画面設計で選択する。

## 6. カラーパレット（コンテンツ領域向け）

* 背景色（画面全体）：`#FAFAFA`（Tailwind: `bg-gray-50`）
* カード背景：`#FFFFFF`（Tailwind: `bg-white`）
* 枠線：`#E5E7EB`（Tailwind: `border-gray-200`）
* 本文色：`#1F2937`（Tailwind: `text-gray-800`）
* サブテキスト：`#6B7280`（Tailwind: `text-gray-500`）

配色は AppHeader / Footer の設計と整合させ、画面ごとに独自色を多用しないこと。

---

## 7. レスポンシブ設計指針

* すべての画面は **モバイルファースト** で設計する。
* スマートフォン（~640px）

  * 基本は 1 カラム構成（縦に積む）。
  * フォーム・カードは幅いっぱい（`w-full`）を基本とする。
* タブレット（768px~）以上

  * コンテンツに応じて 2 カラム以上の `grid` を許容するが、ログイン画面などシンプルな画面では 1 カラムを維持してよい。
* カードレイアウト

  * 一覧系画面では `grid`（例：`grid-cols-1 md:grid-cols-2`）を利用するが、角丸・影・余白のトーンは共通とする。

## 8. コンテンツ領域内の UI ルール

### 8.1 カード構造

* 角丸：`rounded-2xl`
* 影：`shadow-sm`（Hover 時に `shadow-md` までを上限とする）
* 内部余白：`p-4 md:p-6`
* 背景：`bg-white`

ログイン画面など、単一フォームのみを含むカードでは、上記を基本としつつ高さを過度に伸ばさないようにする。

### 8.2 ボタン配置

* 主要操作ボタン（Primary）は、フォーム末尾やカード右下付近に配置する。
* 補助ボタン（Secondary / Link 系）は、主要ボタンの左側または下側に配置し、視覚的階層をはっきり区別する。
* ボタン間隔の基準

  * ボタン同士：`space-x-3`
  * フォーム本体とボタン行：`mt-4`

### 8.3 入力フォーム

* ラベルは原則として入力欄の上（上部ラベル）に配置する。
* 入力欄のベーススタイル（例）

  * 角丸：`rounded-xl`
  * 枠線：`border border-gray-300`
  * 背景：`bg-gray-50`
* フォーカス時は、Tailwind の `focus:outline-none focus:ring-2 focus:ring-primary-500` 等、デザインシステムで定義したスタイルを適用する。
* エラーメッセージ

  * `text-red-500 text-sm mt-1`
  * コンポーネントとしては `InlineFieldError`（C-15）を使用し、ここで定義したスタイルをベースに実装する。

### 8.4 アイコン・画像

* アイコンは線形（outline）を基本とし、`lucide-react` 等の共通アイコンセットを利用する。
* 画像は `max-w-full rounded-xl` を基本とし、カード外にはみ出さないようにする。

### 8.5 多言語対応との関係

* 文言の管理は `StaticI18nProvider` および各詳細設計書で定義された i18n キーに従う。
* 本書では `data-i18n` 属性等のカスタム属性は定義しない。
* コンテンツ領域のレイアウトは言語に依存せず、文字量の違いは折り返し・行間で吸収する方針とする。

## 10. 運用ポリシー

* 新規画面の基本設計・詳細設計を作成する際は、本ファイルで定義したコンテンツ領域のルールを参照すること。
* 画面ごとの設計でコンテンツ領域のスタイルを独自定義しない。例外が必要な場合は、本書を改訂して全体ルールとして反映する。
* Tailwind クラスの粒度での指定は、各詳細設計書に任せるが、

  * 余白・幅・角丸・影のトーンは本書に従うこと。
