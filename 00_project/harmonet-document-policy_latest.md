# HarmoNet Document Policy（文書運用ルール）_latest
HarmoNetプロジェクトにおける文書作成・管理・更新ルールを定義する。

---

## 🎯 目的
この文書は、HarmoNet開発に関わる全てのドキュメントの作成・管理・参照方法を統一し、  
AI三層構造（Claude・タチコマ・Gemini）およびTKDによる共同開発を円滑にすることを目的とする。

---

## 🧭 運用基本原則
1. **完全版更新主義（Full Update Principle）**
   - 文書は差分ではなく常に「完全版」で更新する。
   - 改訂履歴は末尾の ChangeLog で管理する。
   - Claude・Gemini・タチコマ全員が最新版を参照できるようにする。

2. **バージョン明記の徹底**
   - ファイル名末尾に `vX.X` を付与。
   - 文書内メタ情報にも Version / Created / Last Updated を必ず記載。
   - 最新版参照リンクは `_latest.md` に統一。

3. **文書構造の一貫性**
   - 章立ては「目的 → 定義 → 手順 → 運用 → ChangeLog」の順序を基本とする。
   - 章番号の省略は禁止。全ファイルで同一スタイルを維持。

4. **命名規則**
   - `snake_case` を基本とし、英語表記を原則とする。
   - 固有日本語タイトルのみ `HarmoNet_` プレフィックスを許可。
   - 例：`harmonet-admin-feature-requirements_v1.0.md`

5. **承認・更新フロー**
   - TKD：最終承認者（ヒューマンリーダー）
   - Claude：構造・要件整理
   - タチコマ：整合・統合・出力責任者
   - Gemini：監査・解析（BAG解析・品質検証）

---

文書はすべて `/docs/` 以下に格納。  
ディレクトリ構造は `harmonet-docs-directory-definition_v3.0.md` に準拠する。

## 🧩 ドキュメント種別定義

| 種別 | 担当 | 内容 |
|------|------|------|
| **要件定義書** | TKD / Claude | 機能・非機能・追加要件の定義 |
| **設計書** | Claude / タチコマ | 機能詳細・UI・スキーマ・ER設計 |
| **運用仕様書** | タチコマ | 管理手順・環境構成・バックアップ |
| **監査レポート** | Gemini | BAG解析・静的/動的品質検証 |
| **ガイドライン** | タチコマ | チーム方針・標準運用ルール |

---

## 🧠 AI三層ドキュメント連携モデル

TKD（要件入力）
↓
Claude（構造化・設計）
↓
タチコマ（整合・統合）
↓
Gemini（監査・品質検証）


### データ形式の統一
- すべての中間生成物は Markdown (`.md`) 形式とする。
- スキーマや定義情報は Markdown内の **YAMLブロック** で構造化。
- Geminiの解析対象は YAML付き文書のみとする。

---

## 🧩 文書ライフサイクル
1. **新規作成**
   - 担当AIが文書を生成。
   - ファイル名・メタ情報を自動付与。
2. **レビュー**
   - 上位AI（またはTKD）が内容承認。
3. **監査**
   - Geminiが構造と品質を検証。
4. **統合**
   - タチコマが`_latest.md`を更新。
5. **アーカイブ**
   - 旧版は`/99_archive/`へ移動。

---

## 🛠️ メタ情報定義（必須項目）
各文書末尾に以下を記載する。

Created: YYYY-MM-DD
Last Updated: YYYY-MM-DD
Version: X.X
Document ID: HNM-<TYPE>-<DATE>
Supersedes: <旧バージョン名>
